# Lost Deal Taxonomy — تصنيف الصفقات المفقودة

## Purpose — الغرض

Every lost or parked deal is logged with a specific reason. This taxonomy creates a structured dataset for pattern recognition — which sectors lose on price, which on trust, which on timing. Review patterns monthly.

كل صفقة مفقودة أو موقوفة تُسجَّل بسبب محدد. هذا التصنيف يبني قاعدة بيانات منظمة للتعلم — أي القطاعات تُخسر بسبب السعر، أيها بسبب الثقة، أيها بسبب التوقيت. مراجعة الأنماط شهرياً.

---

## How to Log — كيفية التسجيل

For each lost deal, record:
- Company label (no PII)
- Sector
- Deal size (SAR range)
- Primary loss reason (category below)
- Sub-reason (from list below)
- Lesson learned (1-2 sentences)
- 90-day reopen trigger (Yes / No — and the trigger condition)

---

## Category 1 — Price | السعر

The deal was lost primarily because of cost concerns.

| Sub-reason — السبب الفرعي | Description | Reopen Trigger |
|---|---|---|
| P1 — Too expensive overall | Client considered the total project cost too high relative to perceived value | Budget cycle opens / new funding / competitor delivers poorly |
| P2 — Competitor was cheaper | Client chose a lower-cost provider for a comparable scope | Competitor fails to deliver / client returns to reopen |
| P3 — Budget cut mid-process | Budget was available at discovery; internal approval was not granted | Next budget cycle / leadership change |
| P4 — No budget available | Budget never confirmed; prospect had no funding path | Confirm budget allocation in 90 days |

**Lesson pattern for Price losses:** If Price is the top loss category in a sector, review proposal structure — ensure value is presented before price. Review [PRICING_GUARDRAILS.md](PRICING_GUARDRAILS.md) discount rules.

---

## Category 2 — Timing | التوقيت

The deal was real but the timing was wrong.

| Sub-reason — السبب الفرعي | Description | Reopen Trigger |
|---|---|---|
| T1 — Not ready yet | Client acknowledged the need but is not operationally ready to start | Client signals readiness — re-engage in 60–90 days |
| T2 — Internal freeze | Hiring freeze, restructuring, or strategic review blocked new projects | Freeze lifted — monitor via LinkedIn or referral signal |
| T3 — Wrong quarter | Budget available but already committed for this quarter | Start of next budget quarter |
| T4 — Key project in progress | Client is mid-way through a conflicting internal project | Conflicting project completed — re-engage |

**Lesson pattern for Timing losses:** High Timing losses indicate good prospect quality but poor timing — maintain in nurture sequence. Do not close the file.

---

## Category 3 — Competition | المنافسة

The deal was lost to a competing alternative.

| Sub-reason — السبب الفرعي | Description | Reopen Trigger |
|---|---|---|
| C1 — Chose a named competitor | Client selected a competing AI/tech provider | Competitor contract ends / client expresses dissatisfaction |
| C2 — Chose in-house | Client decided to build internally with existing IT team | In-house project stalls or fails — monitor at 6 months |
| C3 — Chose no-AI approach | Client decided to continue with manual processes | Pain increases / new leadership / regulatory driver |
| C4 — Chosen on relationship | Incumbent vendor retained due to existing relationship | Relationship contact changes role |

**Lesson pattern for Competition losses:** Analyze which competitor wins most frequently and in which sector. Does their win pattern reveal a pricing gap or a capability gap?

---

## Category 4 — Fit | الملاءمة

The deal was lost because Dealix was not the right solution for this client.

| Sub-reason — السبب الفرعي | Description | Reopen Trigger |
|---|---|---|
| F1 — Wrong sector or size | Client is outside Dealix's current target sectors or too small/large | Sector expansion or client scales |
| F2 — Data not ready | Client's data infrastructure is too immature to support the proposed solution | Client resolves data infrastructure issues |
| F3 — Use case unclear | Client could not define the specific workflow they wanted improved | Client develops clearer internal requirements |
| F4 — Scope mismatch | Client wanted a different scope than Dealix offers (e.g., pure software, not services) | Client requirements change |

**Lesson pattern for Fit losses:** If Fit losses are frequent in a specific sector, review offer positioning for that sector. Update [../14_DISCOVERY_CALL_TEMPLATE.md](../14_DISCOVERY_CALL_TEMPLATE.md) to qualify Fit earlier.

---

## Category 5 — Trust | الثقة

The deal was lost because the client did not have sufficient confidence in Dealix.

| Sub-reason — السبب الفرعي | Description | Reopen Trigger |
|---|---|---|
| TR1 — New company concern | Client uncomfortable with Dealix's track record or company age | Portfolio grows / referrals increase |
| TR2 — AI skepticism | Client does not believe AI systems will work in their environment | Market maturity / pilot success in their sector |
| TR3 — No sector references | Client wanted references from their specific sector | Sector reference available |
| TR4 — Data security concern | Client was not satisfied with Dealix's data handling assurances | Additional certification or audit completed |

**Lesson pattern for Trust losses:** Prioritize completing a pilot in each new sector. Even one delivered project creates the reference that unlocks Trust-blocked deals.

---

## Category 6 — Process | العملية

The deal was lost due to internal client-side process failure.

| Sub-reason — السبب الفرعي | Description | Reopen Trigger |
|---|---|---|
| PR1 — Decision stalled | No decision was reached despite repeated follow-up — contact became unresponsive | Contact reaches out / new contact identified |
| PR2 — Key contact left | The champion or decision maker left the organization | New contact identified and engaged |
| PR3 — Procurement blocked | Procurement process requirements (vendor registration, long approval cycles) prevented close | Vendor registration completed / procurement process simplified |
| PR4 — Legal/compliance block | Legal team raised concerns that were not resolved | Legal concerns addressed by client team |

**Lesson pattern for Process losses:** If Procurement is frequent, consider adding a vendor pre-qualification step to discovery. Reference [../governance/NDA_TEMPLATE.md](../governance/NDA_TEMPLATE.md) and [../governance/DPA_TEMPLATE.md](../governance/DPA_TEMPLATE.md) to reduce legal friction.

## Monthly Loss Review — المراجعة الشهرية

Founder reviews loss log at end of each month:

1. Count losses by category — which is the top loss driver this month?
2. Count losses by sector — which sector has the highest loss rate?
3. Identify any deal where loss reason suggests a process gap (not a market gap) — fix the process.
4. Flag any lost deals with a 90-day reopen trigger that now qualifies for reactivation.
5. Feed patterns into [OBJECTION_LIBRARY.md](OBJECTION_LIBRARY.md) updates if new objection types emerge.
