# حدود التفاوض (Negotiation Guardrails)

مكتب التفاوض يقترح **استراتيجيات رد وتعديلات نطاق** فقط. لا يلتزم بشيء.

## ما يُسمح للنظام باقتراحه

- تجربة (pilot) أصغر.
- نطاق على مراحل (phased scope).
- تشخيص أولاً (diagnostic first).
- مستوى احتجاز (retainer) أقل.
- جدول زمني بديل.
- سبرنت يثبت القيمة أولاً (proof-first).
- نموذج شراكة.

## ما لا يستطيع النظام فعله أبداً (A3)

| ممنوع | السبب |
|-------|-------|
| `approve_discount` | الخصومات تتطلب موافقة المؤسس |
| `set_final_price` | السعر النهائي قرار بشري |
| `accept_legal_terms` | الشروط القانونية تُراجَع بشرياً |
| `guarantee_results` | لا نَعِد بنتائج مضمونة إطلاقاً |
| `commit_delivery_date_without_capacity_check` | يلزم فحص الطاقة |
| `sign_or_accept_contract` | لا توقيع آلي |
| `commit_payment_terms` | شروط الدفع بموافقة |
| `promise_refund` | قرارات الاسترداد بشرية |

## آلية العمل

1. يُصنَّف الرد (`reply_classifier`).
2. إذا كان الرد قابلاً للتفاوض (اعتراض سعري، ليس الآن، طلب عقد، اهتمام بشراكة،
   طلب تفاصيل) يبني `negotiation_desk` مسودة تحمل:
   - `allowed_response`: الرد المسموح.
   - `forbidden_commitments`: قائمة الالتزامات الممنوعة (دائماً مرفقة).
   - `scope_adjustment_options`: بدائل النطاق.
   - `approval_required=true`.
3. طلب العقد ⇒ **تصعيد للمؤسس** تلقائياً، لا قبول.

كل مسودة تفاوض هي اقتراح بشري المراجعة — لا تُرسَل تلقائياً.
