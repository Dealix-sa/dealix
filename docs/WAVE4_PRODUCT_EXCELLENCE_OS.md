### 2.2 Feature Prioritization Framework (RICE)

**Every new feature scored on RICE**:

- **R**each: How many customers impacted? (1–3 = low, 4–10 = medium, 10+ = high)
- **I**mpact: How much value per customer? (10% improvement = small, 50%+ = massive)
- **C**onfidence: How sure are we this will work? (50–100%)
- **E**ffort: How many weeks to build? (1–4 = small, 5–12 = medium, 12+ = large)

**RICE Score** = (Reach × Impact × Confidence) / Effort

**Priority tiers**:
- Score >50: P0 (build immediately)
- Score 20–50: P1 (build in next sprint)
- Score 10–20: P2 (backlog)
- Score <10: P3 (future/nice-to-have)

**Example**:
```
Feature: "Mobile app for KPI dashboard"
Reach: 8 customers = 8
Impact: 30% time savings = medium (1.5x multiplier)
Confidence: 80% = 0.8
Effort: 6 weeks = medium (divide by ~3)

RICE = (8 × 1.5 × 0.8) / 3 = 9.6
Status: P3 (backlog — nice-to-have, not urgent)
```

## 4. Technology Stack & Architecture

### 4.1 Core Technology Choices

#### **Language & Framework**
- **Backend API**: Python (FastAPI) — speed + NLP libraries
- **Automation/Scripts**: Python (Celery for task queues)
- **Data processing**: SQL (PostgreSQL) + Python (Pandas)
- **Frontend** (future): React or Vue + TypeScript
- **Mobile** (future): React Native (code reuse)

**Decision rationale**:
- Python: Strong AI/ML libraries (transformers, LLMs), fast prototyping, team expertise
- FastAPI: Modern, async, great for integrations
- PostgreSQL: Reliable, PDPL compliance tracking, full-text search for Arabic
- React: Industry standard, hire-able team, component reusability

#### **Hosting Infrastructure**
- **Primary**: Railway.app or AWS (flexibility for scale)
- **Database**: PostgreSQL (managed, backup + replication)
- **Secrets**: HashiCorp Vault (or Railway secrets)
- **CI/CD**: GitHub Actions (built-in, free)
- **Monitoring**: Sentry (error tracking) + CloudWatch (infrastructure)
- **Logging**: Structured JSON logs → CloudWatch or ELK

**Decision rationale**:
- Railway: Saudi-friendly, simple deployment, clear pricing
- PostgreSQL: PDPL-compliant audit trail capability
- GitHub Actions: Integrated with repo, no extra tools
- Sentry + CloudWatch: Visibility into production issues

#### **Data & Security**
- **API authentication**: OAuth 2.0 + JWT tokens
- **Encryption**: TLS 1.3 for transport, at-rest encryption for sensitive data
- **Audit logging**: Every API call logged with user + timestamp + action
- **PDPL compliance**: Data classification per PDPL categories, retention policies
- **Rate limiting**: Per-customer throttling (prevent abuse)

#### **Integration Points**
- **CRM integration**: Zoho CRM API (customer case-study), Salesforce (future)
- **Email**: SendGrid API (transactional), SMTP (customer domain)
- **WhatsApp**: Twilio WhatsApp API (Wave 2 feature)
- **Data export**: CSV/JSON export, scheduled reports

### 4.2 Architecture Principles

**Single Responsibility**:
- Each OS has dedicated modules (WhatsApp Revenue OS ≠ Review Intelligence OS)
- Shared utilities in `common/` (auth, logging, PDPL, integrations)

**API-first**:
- All functionality exposed via REST API
- Frontend consumes same APIs as customer integrations
- No tight coupling between layers

**Testable**:
- Unit tests for business logic (>80% coverage)
- Integration tests for OS-specific workflows
- E2E tests for critical paths (authentication, data export)

**Auditable**:
- Every data change logged (who, what, when, why)
- Immutable transaction log (for PDPL + audit)
- Query traceability (customer data access tracked)

**Scalable**:
- Stateless API servers (horizontal scaling)
- Async job queue for heavy lifting (data processing, exports)
- Caching layer (Redis, future) for KPI dashboards
- Database indexing for common queries

### 4.3 Development Workflow

**Version control**:
- Main branch = production
- Feature branches = `feat/os-name-feature` (e.g., `feat/whatsapp-lead-deduplication`)
- PR review = minimum 1 reviewer (founder or lead engineer)
- Merge = squash commits (clean history)

**Deployment**:
- Git push → GitHub Actions CI/CD pipeline
- Automated tests run (must pass to deploy)
- Manual approval gate (founder or lead engineer)
- Deploy to staging first (24-hour soak test)
- Deploy to production (if no issues)

**Rollback**:
- Maintain previous version in production (blue-green deployment)
- Rollback = 1-click revert to previous version
- RTO (Recovery Time Objective): <30 minutes
- RPO (Recovery Point Objective): <5 minutes

## 5. Quality & Deployment Standards

### 5.1 Definition of Done (DoD)

Every OS feature is "done" only when:

- [ ] Code review passed (≥1 reviewer, founder if critical)
- [ ] Unit tests written + passing (>80% code coverage)
- [ ] Integration tests passing (feature works end-to-end)
- [ ] Security review passed (no SQL injection, API auth, data handling)
- [ ] Performance tested (API response <2 sec, data processing <30 sec)
- [ ] Documentation updated (code comments, API docs, user guide)
- [ ] Deployed to staging environment
- [ ] 24-hour soak test (no errors in staging)
- [ ] Customer notified (changelog, training if needed)

### 5.2 Testing Standards

#### **Unit Tests**
- Test individual functions in isolation
- Mock external dependencies (APIs, databases)
- Target: >80% code coverage
- Framework: pytest (Python)

**Example**:
```python
def test_whatsapp_lead_deduplication():
    # Given: Two identical leads from WhatsApp
    lead1 = Lead(name="Ahmed", phone="+966501234567", source="whatsapp")
    lead2 = Lead(name="Ahmed", phone="+966501234567", source="whatsapp")
    
    # When: Deduplication runs
    result = deduplicate_leads([lead1, lead2])
    
    # Then: Only one lead remains
    assert len(result) == 1
    assert result[0].id == lead1.id  # Keep first, mark second as duplicate
```

#### **Integration Tests**
- Test end-to-end workflow (API → Database → Output)
- Use test database (PostgreSQL)
- Target: >70% critical path coverage
- Framework: pytest + test fixtures

**Example**:
```python
def test_whatsapp_lead_workflow(test_client, test_db):
    # Given: Customer API credentials + WhatsApp webhook
    customer_id = "cust_12345"
    
    # When: WhatsApp lead comes in
    lead_data = {
        "customer_id": customer_id,
        "name": "Salma",
        "phone": "+966501234567",
        "message": "I'm interested in your services"
    }
    response = test_client.post("/api/leads/webhook", json=lead_data)
    
    # Then: Lead stored + deduplicated
    assert response.status_code == 200
    stored_lead = test_db.query(Lead).filter_by(phone="+966501234567").first()
    assert stored_lead is not None
    assert stored_lead.status == "new"
```

#### **E2E (End-to-End) Tests**
- Test full user workflow (as customer would use)
- Exercise UI + API + database + integrations
- Run in staging environment (production-like)
- Target: All critical user paths
- Framework: Selenium (UI) or API-based

**Critical paths to test**:
1. Lead capture → deduplication → assignment
2. KPI calculation → reporting → export
3. System configuration → workflow activation
4. User access → audit logging → data export

#### **Performance Tests**
- Measure API response times (target: <2 sec for most, <30 sec for heavy)
- Load test (simulate N concurrent users, does system degrade gracefully?)
- Database query performance (are queries indexed properly?)
- Tool: Apache JMeter or custom Python script

**Example load test targets**:
```
Single API call response: <500 ms (p95), <2 sec (p99)
Bulk data processing (1000 records): <30 seconds
Concurrent users (100 using system): No 5xx errors
Database query (indexed): <100 ms
Database query (unindexed): Trigger alert for optimization
```

### 5.3 Security Standards

#### **API Security**
- [ ] All endpoints authenticated (OAuth 2.0 or API key)
- [ ] Rate limiting (100 requests/min per customer)
- [ ] Input validation (no SQL injection, XSS, etc.)
- [ ] CORS configured (customer's domain only)
- [ ] HTTPS only (TLS 1.3)
- [ ] Security headers (HSTS, X-Content-Type-Options, etc.)

#### **Data Security**
- [ ] Customer data encrypted at rest (AES-256)
- [ ] PII flagged + extra access controls
- [ ] PDPL audit log (who accessed what data, when)
- [ ] Data retention policies enforced (delete old data on schedule)
- [ ] Backup testing (monthly restore drills)

#### **Code Security**
- [ ] Dependencies scanned for vulnerabilities (Dependabot)
- [ ] Secrets (API keys) never in code (use environment variables)
- [ ] No hardcoded test data with real information
- [ ] SQL parameterization (prevent injection)

#### **PDPL Compliance Checklist**
- [ ] Customer consent collected + documented
- [ ] Data classification recorded (personal, sensitive, public)
- [ ] Access logs maintained (audit trail)
- [ ] Data deletion honored (<30 days)
- [ ] Data portability available (export in standard format)
- [ ] Security incident response plan documented

### 5.5 Deployment Checklist

**Before deploying to production**:

```
PRE-DEPLOYMENT
[ ] All tests passing (unit, integration, E2E)
[ ] Code review approved (≥1 reviewer)
[ ] Security scan passed (no vulnerabilities)
[ ] Performance test passed (meets latency targets)
[ ] Database migrations tested (data integrity intact)
[ ] Rollback plan documented (previous version identified)
[ ] Customer notified (changelog + training if needed)
[ ] Support team briefed (new features, known issues)

STAGING DEPLOYMENT
[ ] Deploy to staging environment
[ ] Smoke tests pass (critical paths work)
[ ] 24-hour soak test (no errors, no performance degradation)
[ ] Approval from founder/tech lead

PRODUCTION DEPLOYMENT
[ ] Blue-green deployment (new version runs alongside old)
[ ] Traffic gradually shifted to new version (10% → 50% → 100%)
[ ] Monitor error rates + performance (first 1 hour)
[ ] If issues detected: Instant rollback to previous version
[ ] Post-deployment verification (all systems healthy)
[ ] Team notification (deployment complete, monitoring active)

POST-DEPLOYMENT
[ ] Monitor for 24 hours (extra vigilance)
[ ] Customer feedback solicited (issues? concerns?)
[ ] Documentation updated (if needed)
[ ] Incident report if issues (what went wrong, how to prevent)
```
