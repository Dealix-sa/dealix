# Delivery Daily Rhythm — الإيقاع اليومي للتسليم

> Purpose — الغرض: the daily cadence during an active Command Sprint. It keeps delivery evidence-forward and governed: a morning setup, a live delivery log, evidence capture as you work, and governance checkpoints so nothing external moves without approval.
>
> الإيقاع اليومي أثناء سبرنت قيادة نشط. يبقي التسليم قائمًا على الأدلة ومحكومًا: إعداد صباحي، سجل تسليم حي، التقاط الأدلة أثناء العمل، ونقاط تفتيش حوكمية حتى لا يتحرك أي شيء خارجي بلا موافقة.

## Morning setup — الإعداد الصباحي

Start each delivery day the same way:

- Open the customer workspace `customers/<name>/` and the delivery log `09_delivery_log.md`.
- Restate the single outcome the sprint is proving. If today's work does not move that outcome, cut it.
- Check the approval queue for any draft from yesterday awaiting founder sign-off.
- Confirm you are working only on passported data.

ابدأ كل يوم تسليم بالطريقة نفسها: افتح مساحة عمل العميل وسجل التسليم؛ أعِد ذكر النتيجة الواحدة؛ راجع طابور الموافقات لأي مسودة من الأمس؛ تأكد أنك تعمل على بيانات ذات جواز مصدر فقط.

---

## The delivery log — سجل التسليم

`09_delivery_log.md` is the running record of the sprint. Append, never rewrite. Each entry is dated and carries:

- What was done today, in one or two lines.
- What was observed (numbers, screenshots references, before/after).
- Any limitation or caveat discovered.
- Any draft produced and its approval status.

سجل التسليم `09_delivery_log.md` هو السجل الجاري للسبرنت. أضِف ولا تُعِد الكتابة. كل قيد مؤرَّخ ويحمل: ما أُنجِز اليوم؛ ما لوحِظ؛ أي قيد اكتُشِف؛ أي مسودة وحالة موافقتها.

---

## Evidence capture — التقاط الأدلة

Evidence that is not captured does not exist. Capture as you work, not at the end:

- Record before/after states the moment you change something.
- Reference artifacts by path, not by retelling.
- Note the value tier of each observation (estimated / observed / verified / client_confirmed) so the proof pack inherits honest labels.
- Never paste raw PII into the log; use anonymized labels.

الدليل غير الملتقَط غير موجود. التقِط أثناء العمل لا في النهاية: سجّل حالة قبل/بعد لحظة التغيير؛ أشِر إلى النواتج بالمسار؛ دوّن طبقة قيمة كل ملاحظة؛ لا تلصق بيانات شخصية خام — استخدم تسميات مجهّلة.

---

## Governance checkpoints — نقاط التفتيش الحوكمية

Three checkpoints run every delivery day:

1. **Draft gate** — every external-facing artifact is created as `draft_only` and enters the approval queue. Nothing is sent without a logged founder approval.
2. **PII gate** — any artifact that would expose PII outside the workspace requires explicit approval before it leaves.
3. **Claim gate** — no deliverable states a guaranteed return, conversion rate, or ROI as fact. Use estimated / observed / verified language.

ثلاث نقاط تفتيش يوميًا: بوابة المسودة (كل ناتج خارجي مسودة في طابور الموافقة)؛ بوابة البيانات الشخصية (أي كشف خارجي يتطلب موافقة)؛ بوابة الادعاء (لا عائد مضمون ولا نسبة تحويل كحقيقة).

---

## End of day — نهاية اليوم

- Append the final delivery-log entry for the day.
- Move any completed draft into the approval queue for founder review.
- Note tomorrow's single most important step.
- If a proof event occurred, it is logged so the Revenue Command Center reflects it.

في نهاية اليوم: أضِف القيد الأخير للسجل؛ انقل أي مسودة مكتملة إلى طابور الموافقة؛ دوّن خطوة الغد الأهم؛ سجّل أي حدث إثبات وقع.
