# DEALIX Sales Deck Guide

**Owner:** Founder
**Source of truth:** `docs/brand/DEALIX_MARKETING_ASSET_GUIDE.md` + `docs/positioning/`

## Deck purpose

A Dealix deck moves a qualified buyer from "I want to understand what this is" to "I want to run a Diagnostic." It does not close a sprint by itself. The sprint closes on the Diagnostic.

## Structure — 12 slides

1. Cover
2. The category — Revenue Operating System
3. The problem — three failure modes Saudi B2B operators face
4. The pattern — what changes when revenue is run as an operating system
5. What Dealix does — three sentences, three pillars
6. What Dealix does NOT do — explicit exclusions
7. Trust architecture — gates, approval classes, founder sign-off
8. Sprint catalog — list of named sprints with price ranges
9. Proof model — what evidence Dealix produces (anonymized example)
10. Saudi context — PDPL, CITC, Arabic-first, local impact
11. Engagement path — Diagnostic, Sprint, Retainer
12. Next step — Diagnostic offer + price + start window

12 slides is the cap. Speakers who need 15 slides should cut, not add.

## Slide-by-slide content

### 1. Cover

- Lockup centered.
- Customer name + date in the lower-third.
- Pipeline gradient backdrop.
- No body copy.

### 2. The category

- H2: "Revenue Operating System"
- One paragraph: define the category (3 sentences).
- One visual: the four-layer stack (Deal Flow / Trust Gates / Approved Execution / Capital Loop).

### 3. The problem

- H2: "Three failure modes."
- Three named failure modes with one-line description each:
  - Leaky deal flow
  - Ungoverned AI execution
  - Disconnected proof and capital

### 4. The pattern

- H2: "What changes when revenue runs as an OS."
- Three bullets, each a state shift: "From X to Y."

### 5. What Dealix does

- H2: "What Dealix does."
- Three sentences. Three pillars.
  - Intelligent deal flow
  - Founder-approved growth
  - Trust-gated AI execution

### 6. What Dealix does NOT do

- H2: "What Dealix does not do."
- Five explicit exclusions:
  - Not scraping.
  - Not bulk cold outreach.
  - Not WhatsApp automation.
  - Not a CRM.
  - Not a generic agency.

### 7. Trust architecture

- H2: "Approval gates."
- Table: Approval class (A0-A3) + Example action + Approver.

### 8. Sprint catalog

- H2: "Sprints we run."
- Table: Sprint name + Length (days) + Price range (SAR) + Output.
- Footnote: "Pricing is range-based. Diagnostic confirms the line item."

### 9. Proof model

- H2: "Proof model."
- Anonymized sprint outcome. Dated. Sector named, customer not.
- Footnote disclaimer: "Estimated value is not Verified value."

### 10. Saudi context

- H2: "Built for Saudi B2B."
- Three points:
  - PDPL- and CITC-aware.
  - Arabic-first or bilingual.
  - Founder-buyer focused.

### 11. Engagement path

- H2: "How we work together."
- Three-stage visual: Diagnostic → Sprint → Retainer.
- Each stage: input, output, cost band, gate.

### 12. Next step

- H2: "Next step: Diagnostic."
- Price: stated in SAR.
- Window: "Available start dates within 14 days."
- One CTA: "Confirm and I'll send the Diagnostic Proposal."
- Founder signature line.

## Speaker notes

- Founder-voice, first person.
- Each slide has 60-90 words of notes.
- Each slide has one stated objection and the prepared response.
- Cross-link to `docs/positioning/OBJECTION_HANDLING.md`.

## Variants

| Variant | Slides changed |
|---|---|
| Sector-tailored | Slide 3, 4, 9 use sector-specific examples |
| Partner pitch | Slide 11 adds a partner economics line |
| Investor brief | Slide 8 adds a unit-economics line; slide 12 replaces CTA with raise terms |

The investor variant is a separate deck, not a sales deck. Use the sales deck for sales only.

## Failure mode

- Deck balloons to 18 slides.
- A "Why Dealix" slide replaces "What Dealix does NOT do."
- Slide 9 cites a verified outcome without source.
- Speaker notes contain a guarantee phrase.

## Recovery path

1. Cut back to 12 slides.
2. Restore the exclusions slide.
3. Re-source slide 9 metrics to a Proof Pack reference.
4. Re-run the voice checklist on speaker notes.
