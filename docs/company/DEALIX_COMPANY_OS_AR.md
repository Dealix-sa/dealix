# ديالكس — الشركة كنظام تشغيل

> **المصدر المرجعي لبنية شركة ديالكس ونظام تشغيلها.**
> الفرع: `phase/startup-architecture-brand-os`
> الحالة: وثيقة حيّة. كل التواصل الخارجي مسودة فقط افتراضيًا (`OUTBOUND_MODE=draft_only`).
> لغة المرجع: العربية هي لغة التموضع الأولى للشركة. النسخة الإنجليزية في `DEALIX_COMPANY_OS_EN.md`.

---

## 1. الرؤية

أن نجعل كل شركة سعودية تعمل بنظام تشغيل ذكي — لا بأدوات متفرقة، ولا بوكالة خارجية، ولا بلوحة عاجزة عن التنفيذ.

ديالكس لا تبيع شات بوت، ولا CRM، ولا لوحة تحكم، ولا خدمة وكالة. ديالكس تصمّم وتبني وتُشغّل أنظمة تشغيل ذكية مُحكمة — واحد لكل وظيفة تجارية — تركب داخل الشركات السعودية الصغيرة والمتوسطة وتُدير تلك الوظيفة من الاكتشاف إلى القرار إلى التسليم إلى الإثبات.

التموضع سعودي أصيل: شركة ذات مسؤولية محدودة (LLC) سعودية، تسعير بالريال السعودي، نطاق زمني آسيا/الرياض، التزام بنظام حماية البيانات الشخصية (PDPL)، ومتابعة لمتطلبات هيئة الزكاة والضريبة والغرفة التجارية. اللغة الأولى هي العربية، مع مرايا إنجليزية للمستثمرين والشركاء الدوليين.

مبدأ التشغيل:

> الذكاء الاصطناعي يستكشف ويحلّل ويوصي. العمليات الحاسمة تنفّذ. الإنسان يعتمد الالتزامات الخارجية الحرجة.
> لا إرسال تلقائي للخارج. لا عملاء وهميون، لا شهادات ملفقة، لا ضمانات بعائد محدد. كل التوقعات سيناريوهات بمستوى ثقة وفرضيات صريحة.

---

## 2. المهمة

نُحوّل كل وظيفة تجارية في الشركة السعودية من فوضى يدوية إلى نظام تشغيل ذكي قابل للقياس والإثبات — خلال أسابيع، لا سنوات.

المهمة تشغيلية وليست إنشائية: نأخذ وظيفة موجودة (الإيرادات، التسليم، المتابعة، القرار)، نبني لها نظامًا صغيرًا، نُشغّله، ثم نُثبت الفرق ببيانات قبل/بعد موثّقة.

---

## 3. المنتجات (14 منتج)

كل منتج هو نظام تشغيل مستقل: مدخلات محددة، مخرجات محددة، بوابات اعتماد، وحزم إثبات.

| # | المنتج | الوظيفة |
|---|---|---|
| 1 | غرفة قيادة الإيرادات | قيادة الإيرادات اليومية: الأنابيب، مسودات التواصل، ملخص المدير، المؤشرات. |
| 2 | دماغ الشركة | طبقة المعرفة والذاكرة المركزية: القرارات، أدلة التشغيل، سياق العملاء. |
| 3 | نظام متابعة الواتساب والبريد | مسودات متابعة معتمدة عبر الواتساب والبريد. لا إرسال حي افتراضيًا. |
| 4 | نظام مراجعة بريد التواصل | مسودات بريد خارجي، طابور مراجعة، اعتماد قبل أي إرسال. |
| 5 | نظام إشعارات ومتابعة SMS | مسودات إشعالات ومتابعات SMS؛ `SMS_SEND_ENABLED=false` افتراضيًا. |
| 6 | نظام الثقة والامتثال بالذكاء الاصطناعي | ضوابط PDPL، فئات الاعتماد، سجلات التدقيق، سجل الادعاءات الممنوعة، حزم الإثبات. |
| 7 | نظام تسليم العملاء | اكتشاف ← بناء ← اختبار ← تقرير نجاح لكل عميل. |
| 8 | نظام التواصل الحي المُحكم | تواصل حي خلف بوابات صارمة؛ يتطلب تفعيل الأعلام. افتراضي معطّل. |
| 9 | مكتب قرارات المؤسس | طابور القرارات اليومي، مصفوفة التصعيد، مصفوفة التفويض. |
| 10 | سبرنت تشخيص الشركة | تشخيص مدفوع لأسبوع واحد: رسم الألم، تدقيق البيانات، الخط الأساسي، التوصية. |
| 11 | نظام ذكاء العروض | سُلّم العروض، محرك التسعير، توليد عروض قطاعية. |
| 12 | نظام مراقبة السوق والمنافسين | رصد القطاعات، إشارات المنافسين، تحولات السوق، تحديثات تنظيمية سعودية. |
| 13 | نظام العروض والعقود | توليد العروض والعقود، فواتير متوافقة مع زاتكا، روابط دفع موياسار. |
| 14 | نظام حزم الإثبات التنفيذية | حزم إثبات قبل ← بعد ← فرق موثّق للتنفيذيين. |

المنتجات الأربعة عشر تشترك في نفس محرك الثقة، نفس بوابات الاعتماد، ونفس وضعية "مسودة فقط افتراضيًا" للتواصل الخارجي.

## 4. العمليات اليومية

### الصباح (30 دقيقة) — غرفة القيادة
1. `make company-day`
2. مراجعة `reports/command_room/index.html`
3. مراجعة المسودات في `outbox/YYYY-MM-DD/`
4. مراجعة طابور اعتماد المؤسس في مكتب قرارات المؤسس

### أثناء اليوم
5. اعتماد / رفض / إعادة صياغة / اختصار / جعل رسمي / تغيير العرض / نقل للتنشئة / عدم التواصل — لكل مسودة.
6. الإرسال اليدوي للمسودات المعتمدة فقط (ويشترط تفعيل `EXTERNAL_SEND_ENABLED` + أعلام القناة).
7. تحديث `ledgers/outreach_log.csv` بعد كل إرسال.
8. تسجيل الردود في `ledgers/reply_log.csv`.

### المساء (15 دقيقة)
9. تحديث `ledgers/deals_pipeline.csv`.
10. مراجعة `reports/revenue/YYYY-MM-DD/daily_ceo_report.md`.
11. تسجيل القرارات في دماغ الشركة.

### القاعدة الذهبية
لا تُرسل أي شيء للخارج دون اعتماد إنساني صريح وتفعيل الأعلام المناسبة. الحالة الافتراضية للنظام بأكمله هي: توليد مسودات + مراجعة بشرية.

## 5. الأدوار

### اليوم (مرحلة المؤسس)
- **المؤسس / الرئيس التنفيذي / المُشغّل** — يدير الحلقة التجارية اليومية، الاعتمادات، التسليم، قرارات المؤسس.
- **الذكاء الاصطناعي / المنصة (قيادة المؤسس هندسيًا)** — بوابة LLM، الوكلاء، أنظمة التشغيل، بناء التسليم.

### أدوار مخططة (توظيف بمشغل، لا بالتوقع)
| الدور | المشغل | مستوى الثقة |
|---|---|---|
| قائد التسليم | تطبيقان متزامنان مدفوعان | عالٍ |
| مساعد مبيعات / مؤسس | اشتراك متكرر واحد بعد التجريبي | متوسط |
| مهندس ثانٍ | 3 بناءات عملاء متزامنة | متوسط |
| قائد امتثال / عمليات | أول تطبيق تجريبي للمؤسسات أو لقاء منظّم | متوسط |
| نجاح العميل | 5 اشتراكات نشطة | متوسط |

التوظيف محافظ ويحكمه مشغل حقيقي. لا توظيف على التوقع وحده.

## 6. اتخاذ القرار

### قاعدة القرار
كل قرار خارجي (إرسال، عرض، عقد، دفع، التزام عميل) يتطلب اعتمادًا إنسانيًا. الذكاء الاصطناعي يوصي فقط.

### مصفوفة الاعتماد
| نوع القرار | من يعتمد | الشرط |
|---|---|---|
| إرسال مسودة تواصل | المؤسس | اعتماد صريح + أعلام القناة مفعّلة |
| عرض سعر لعميل | المؤسس | عبر نظام ذكاء العروض + نظام العروض والعقود |
| عقد عميل | المؤسس | مراجعة قانونية + توقيع |
| تفعيل دفع حي (موياسار) | المؤسس | `MOYASAR_LIVE_MODE=true` + مراجعة |
| تفعيل تواصل حي | المؤسس | `EXTERNAL_SEND_ENABLED=true` + أعلام القناة + opt-in + مراجعة قانونية |
| ادعاء نتائج خارجي | المؤسس | حزمة إثبات حقيقية: قبل ← بعد ← فرق موثّق |
| توظيف | المؤسس | المشغل حاصل فعليًا |

### سجل الادعاءات الممنوعة
يحظر تمامًا:
- ضمان عائد محدد (ROI)
- نسب مئوية محددة كضمان
- عملاء أو شهادات وهمية
- إرسال تلقائي للخارج دون اعتماد

كل توقع خارجي يجب أن يحمل: مستوى ثقة + فرضيات صريحة + وصفه كسيناريو وليس كضمان.

## 7. المقاييس

### النجم الشمالي
- الإيراد المتكرر الشهري (ريال سعودي) من الاشتراكات النشطة.

### مقاييس يومية
- الفرص المُبحَثة
- مسودات التواصل المولّدة (لم تُرسل)
- المسودات المعتمدة
- المسودات المُرسلة يدويًا (فقط إن كانت الأعلام مفعّلة)
- الردود المستلمة
- تحديثات الأنابيب

### مقاييس أسبوعية
- التشخيصات المحجوزة
- التجريبيات المبدوءة
- التجريبيات المكتملة
- الاشتراكات الموقّعة
- حزم الإثبات المُسلّمة

### مقاييس شهرية
- الإيراد المتكرر الشهري (ريال)
- عدد الاشتراكات النشطة
- تشخيصات صافية جديدة
- معدل تحويل التجريبي → اشتراك
- التسرّب
- إجمالي الهامش
- مدة التشغيل (أشهر)

### مقاييس الثقة (دائمًا)
- بوابات الاعتماد مُخترَقة أم لا (الهدف: صفر اختراق)
- أعلام الإرسال في الحالة الصحيحة لكل بيئة
- ضوابط PDPL نشطة
- مخالفات سجل الادعاءات الممنوعة (الهدف: صفر)
- اكتمال سجل التدقيق (الهدف: 100%)

## 8. النموذج التجاري (ملخص)

| المرحلة | السعر | المخرج |
|---|---|---|
| تشخيص | 4,999 ريال | تقرير تشخيصي + توصية بحزمة OS |
| تجريبي (شهر) | 14,999 ريال | نظام OS مبني + حزمة إثبات |
| اشتراك شهري | سيناريو (انظر الجدول) | تشغيل مستمر للأجهزة المعمارية |

سيناريوهات الاشتراك الشهري (توقعات تخطيط، ليست التزام سعر):

| السيناريو | الاشتراك الشهري (ريال) | مستوى الثقة | الفرضيات |
|---|---|---|---|
| المحافظ | 7,500 – 12,000 | متوسط-عالٍ | OS واحد، وظيفة واحدة، عمليات خفيفة |
| الأساسي | 12,000 – 25,000 | متوسط | 2–3 أنظمة مجمّعة، حمل تسليم متوسط |
| الممتد | 25,000 – 60,000 | متوسط-منخفض | حزمة متعددة + تواصل حي محكم + حزم إثبات |

التسعير الفعلي يُحدّد لكل عميل عبر نظام ذكاء العروض ويُؤكَّد في نظام العروض والعقود.

## 9. دورة حياة العميل

```
بحث الفرص ← تشخيص ← تجريبي ← اشتراك ← توسّع ← تجديد
```

- **بحث الفرص:** فرص سعودية B2B، تسجيل ICP، إثراء واعٍ بـ PDPL.
- **تشخيص:** أسبوع مدفوع، رسم ألم، تدقيق بيانات، خط أساسي.
- **تجريبي:** 4 أسابيع — اكتشاف ← بناء ← اختبار ← تقرير نجاح.
- **اشتراك:** تشغيل شهري للنظام بعد نجاح التجريبي.
- **توسّع:** إضافة 2–3 أنظمة مجمّعة باستخدام نفس محرك الثقة.
- **تجديد:** حزمة إثبات تنفيذية هي وثيقة التجديد.

## 10. النشر والامتثال

- **PDPL أصيل:** طلبات أصحاب البيانات، استجابة الخرق، سياسة الاحتفاظ، ملحق النقل العابر للحدود.
- **زاتكا:** جاهزية الفوترة الإلكترونية، موياسار وضع الاختبار افتراضيًا.
- **الاعتماد أولًا:** لا التزامات خارجية ذاتية. الإنسان يعتمد.
- **أعلام الإرسال:** كلها معطّلة افتراضيًا. لا إرسال للخارج دون تفعيل صريح لكل بيئة.

أعلام التواصل الخارجي (الحالة الافتراضية لكل بيئة):
- `EXTERNAL_SEND_ENABLED=false`
- `EMAIL_SEND_ENABLED=false`
- `WHATSAPP_SEND_ENABLED=false`
- `WHATSAPP_ALLOW_LIVE_SEND=false`
- `SMS_SEND_ENABLED=false`
- `OUTBOUND_MODE=draft_only`
