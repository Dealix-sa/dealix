# سياسة بوابة بروتوكول سياق النموذج (MCP Gateway Policy)

تحدد هذه السياسة الضوابط الفنية والأمنية الحاكمة لاستخدام وتكامل بروتوكول سياق النموذج (Model Context Protocol - MCP) لربط وكلاء الذكاء الاصطناعي بالأدوات وقواعد البيانات الخارجية، بهدف توفير أقصى درجات الأمان وحماية النظم من الاختراق وتسميم البيانات.

---

## 1. التهديدات والثغرات المستهدفة بالدفاع (Threat Vectors Addressed)
تستهدف هذه السياسة صد وتخفيف المخاطر التقنية التالية المرتبطة بـ MCP:
* **تسميم الأدوات (Tool Poisoning)**: محاولة دمج تعليمات خبيثة أو نصوص برمجية معادية داخل بيانات تعريف الأداة (Tool Metadata) لخداع النموذج وتجاوز الفحص.
* **حقن التعليمات غير المباشر (Indirect Prompt Injection)**: استغلال قيام الوكيل بقراءة بيانات خارجية غير موثوقة (مثل رسائل مبيعات مستلمة أو تعليقات عملاء) لتنفيذ تعليمات مخفية داخل النص.
* **تجاوز الصلاحيات (Privilege Escalation)**: استغلال أداة ذات مستوى منخفض لتوليد استدعاء لأداة حساسة خارج حدود تصريح الوكيل.

---

## 2. آليات الدفاع متعددة الطبقات (Multi-layered Defenses)

```text
       [ External Tool Request ]
                   │
                   ▼
┌──────────────────────────────────────┐
│  Layer 1: Metadata Sanitization      │  <- فحص النصوص وتجريد الرموز الخبيثة
└──────────────────┬───────────────────┘
                   ▼
┌──────────────────────────────────────┐
│  Layer 2: Allowlist Validation       │  <- مطابقة الأداة مع قائمة المصرح به للوكيل
└──────────────────┬───────────────────┘
                   ▼
┌──────────────────────────────────────┐
│  Layer 3: Human Approval Grid        │  <- تفعيل طلب الموافقة البشرية للأدوات الحساسة
└──────────────────┬───────────────────┘
                   ▼
       [ Authorized Execution ]
```

---

## 3. معايير بطاقة تعريف الأداة (Tool Card Structure)
يجب تسجيل كل أداة مدمجة مع MCP في سجل الأدوات المصرح به (`tools.json`) ببطاقة تحتوي على التالي:
* `tool_id`: اسم معرف فريد للأداة (مثل `crm_send_email`).
* `risk_level`: مستوى خطورة الأداة (L1 إلى L4).
* `requires_human_approval`: قيمة منطقية تعين ما إذا كان استدعاء الأداة يتطلب موافقة بشرية مسبقة.
* `can_access_pii`: تحديد ما إذا كانت الأداة قادرة على قراءة أو كتابة بيانات التعريف الشخصية.
* `can_write_external`: تعيين ما إذا كانت الأداة قادرة على إرسال بيانات خارج الحدود التقنية للشركة.
* `max_daily_actions`: الحد الأقصى لعدد المرات المسموح باستدعاء الأداة فيه يومياً لتجنب الاستغلال المكثف.

---

## 4. سياسات وضوابط بوابة الـ MCP التشغيلية
1. **القائمة البيضاء الإلزامية (Strict Allowlisting)**: يمنع منعاً باتاً استدعاء أي أداة أو واجهة برمجة تطبيقات (API) ما لم تكن مسجلة مسبقاً في قائمة الأدوات المعتمدة ومربوطة بالوكيل المعين.
2. **عزل الصلاحيات (Least Privilege)**: يُعطى الوكيل فقط الأدوات والـ APIs اللازمة والضرورية لأداء مهمته الحالية، ويُمنع من الوصول لأي أدوات إدارة نظام أو كتابة قواعد البيانات الكلية.
3. **التدقيق المباشر (Live Intrusion Detection)**: يقوم النظام بفحص مخرجات استعلامات الأدوات بشكل مستمر لكشف أي نصوص تحوي أنماطاً شائعة لهجمات الـ prompt injection أو توجيه النموذج لإلغاء قيود الأمان.
4. **سجل تتبع الأدوات (Tool Execution Audit Log)**: يتم تدوين وتخزين كل عملية استدعاء أداة ناجحة أو فاشلة، متضمنة مدخلات النموذج البرمجي، الرمز البريدي للوكيل، والتوقيع البشري المعتمد، ليكون جزءاً أصيلاً من حزمة الإثبات (Proof Pack) المقدمة للعميل.
