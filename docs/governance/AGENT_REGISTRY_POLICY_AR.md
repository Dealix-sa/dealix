# سياسة تسجيل وحوكمة وكلاء الذكاء الاصطناعي (Agent Registry Policy)

تحدد هذه السياسة الإطار التنظيمي الواجب اتباعه داخل شركة ديلكس وعند تنفيذ مشاريع العملاء لتسجيل وإدارة ومراقبة وكلاء الذكاء الاصطناعي التوليدي (AI Agents) لضمان الشفافية، الأمن، والامتثال المطلق للأنظمة السعودية.

---

## 1. الأهداف الاستراتيجية (Policy Objectives)
* **تحديد الهوية والمسؤولية**: ضمان أن كل وكيل ذكاء اصطناعي نشط مسجل بهوية فريدة، مالك بشري مسؤول، ومهمة تشغيلية محددة بوضوح.
* **إدارة درجات المخاطرة**: تصنيف الوكلاء بناءً على مستوى وصولهم للبيانات وقدرتهم على اتخاذ قرارات مستقلة.
* **الالتزام بأخلاقيات الذكاء الاصطناعي (SDAIA)**: التأكد من مطابقة الوكيل لمبادئ العدالة، الشفافية، المساءلة، والأمان.

---

## 2. مستويات المخاطر للوكلاء (Agent Risk Classification)

يتم تصنيف كل وكيل ذكاء اصطناعي فور إنشائه إلى أحد المستويات الأربعة التالية:

| مستوى المخاطرة | اسم التصنيف | وصف الصلاحيات والوصول | ضوابط الحوكمة المطلوبة |
| :--- | :--- | :--- | :--- |
| **Level 1 (L1)** | **منخفض جداً (Minimal)** | وكلاء للقراءة فقط، لا يتواصلون مع أطراف خارجية، ولا يعالجون بيانات PII حساسة. مثل: وكيل رصد الأخبار العام. | تسجيل بسيط، مراقبة دورية بدون إشعارات موافقة. |
| **Level 2 (L2)** | **متوسط (Moderate)** | وكلاء يولدون مسودات تواصل خارجي أو تقارير عامة، لكن لا يرسلونها للعملاء مباشرة دون مراجعة بشرية. | تسجيل تفصيلي، تفعيل بوابة الموافقات البشرية لجميع المخرجات. |
| **Level 3 (L3)** | **عالي (High)** | وكلاء لديهم صلاحية الكتابة في أنظمة داخلية (مثل CRM / قواعد بيانات الشحن) أو إرسال بريد إلكتروني مباشر مصدق. | تسجيل كامل للوكيل والأدوات، تدقيق فوري، حد يومي لعدد الإجراءات (Rate Limits). |
| **Level 4 (L4)** | **حرج (Critical)** | وكلاء يمكنهم اتخاذ قرارات مالية (مثل استرجاع مبالغ، معالجة مدفوعات) أو الوصول لبيانات صحية أو أمنية حساسة جداً. | **حظر تام للتشغيل المستقل**. تتطلب موافقة بشرية ثنائية (Multi-signature) وسجل أدلة مشفر وغير قابل للتلاعب. |

---

## 3. متطلبات بطاقة تسجيل الوكيل (Agent Registration Card)
يجب تسجيل كل وكيل في سجل الوكلاء (`agents.json`) متضمناً الحقول التالية:
1. `agent_id`: رقم معرف فريد فريد.
2. `name`: اسم الوكيل (مثال: `sales_assistant`).
3. `owner_email`: البريد الإلكتروني للمسؤول البشري عن الوكيل (المالك التشغيلي).
4. `risk_level`: مستوى المخاطرة المعتمد (L1 إلى L4).
5. `allowed_tools`: قائمة بالمعرفات الخاصة بالأدوات والـ APIs التي يُسمح للوكيل باستدعائها.
6. `status`: حالة الوكيل التشغيلية (`active`, `paused`, `deprecated`).
7. `created_at`: تاريخ الإنشاء والتفعيل بالصيغة الدولية ISO 8601.

---

## 4. سياسة الامتثال والمساءلة (Compliance & Audit)
* **المراجعة الدورية**: يتم فحص سجل الوكلاء بشكل نصف شهري لمطابقة المهام الفعلية مع التصاريح المسجلة.
* **إيقاف الطوارئ (Kill Switch)**: يمتلك مدير التشغيل (Sami) الصلاحية المطلقة لإيقاف أي وكيل فوراً عبر الـ CLI عند اكتشاف سلوك شاذ أو انتهاك لضوابط الأمان.
* **الاحتفاظ بالبيانات**: يتم الاحتفاظ بسجلات أنشطة الوكلاء لمدة لا تقل عن 12 شهراً لتلبية متطلبات التدقيق القانوني بموجب نظام PDPL.
