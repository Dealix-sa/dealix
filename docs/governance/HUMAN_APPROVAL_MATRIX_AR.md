# مصفوفة وبوابة الموافقات البشرية (Human Approval Matrix)

تحدد هذه المصفوفة الشروط والإجراءات الواجب توافرها لضمان بقاء الإنسان كعنصر تفعيل وسيط (Human-in-the-loop) في سلسلة قرارات وإجراءات وكلاء الذكاء الاصطناعي، لتفادي اتخاذ قرارات مستقلة قد تضر بالعمل ماليًا أو تشغيليًا أو قانونيًا.

---

## 1. محفزات طلب الموافقة البشرية (Approval Triggers)
يجب إيقاف تنفيذ عمل الوكيل البرمجي فوراً وتوليد طلب موافقة بشرية (Approval Request) في سجل الموافقات (`approvals.json`) في الحالات التالية:
1. **تعديل بيانات الحسابات والعملاء (CRM Modifies)**: محاولة كتابة أو إرسال تحديث لمعلومات العميل الفنية والمالية.
2. **التواصل الخارجي المباشر (External Communication)**: محاولة إرسال بريد إلكتروني، رسالة تواصل، أو إشعارات واتساب حية لشريك أو عميل محتمل.
3. **الوصول للمعلومات الحساسة (PII Access)**: محاولة تصدير أو مشاركة قوائم عملاء تحتوي على هواتف، بريد إلكتروني، أو هويات تنظيمية.
4. **تجاوز مؤشرات المخاطرة (Risk threshold)**: تسجيل الوكيل لدرجة مخاطرة أعلى من القيمة المقررة له في سجل الوكلاء الفعلي.

---

## 2. مصفوفة مستويات الاعتماد (Escalation & Levels)

توزع الصلاحيات والموافقات وفقاً للمصفوفة التالية:

| نوع الإجراء المطلق | مستوى خطورة العملية | المالك المعتمد للموافقة | وقت الاستجابة الأقصى (SLA) | الإجراء البديل عند عدم الرد |
| :--- | :--- | :--- | :--- | :--- |
| **إقرار مسودة تواصل للعميل B2B** | خطورة منخفضة (L2) | مسؤول المبيعات (Sales Rep) / سامي | 24 ساعة عمل | إيقاف الإجراء وتذكير المالك |
| **تسجيل وكيل أو أداة جديدة في النظام** | خطورة متوسطة (L3) | مهندس الحوكمة / سامي | 12 ساعة عمل | رفض التسجيل تلقائياً وتجميد الطلب |
| **تصحيح بيانات شحن وعنوان وطني حي** | خطورة متوسطة (L3) | مسؤول اللوجستيات / متخذ القرار | 4 ساعات عمل | تمرير للتأكيد اليدوي أو إرسال تنبيه واتساب |
| **تصدير سجل أدلة أو بيانات PII عملاء** | خطورة عالية (L3) | مدير الأمن / سامي | 2 ساعة عمل | إلغاء فوري وتوليد تنبيه إيقاف طوارئ |
| **إجراء مالي أو استرجاع نقدي مستقل** | خطورة حرجة (L4) | الرئيس التنفيذي (Sami) | 1 ساعة عمل | رفض فوري وتجميد الحساب احتياطياً |

---

## 3. هيكل طلب الموافقة التشغيلي (Approval Request Schema)
كل طلب موافقة يتم إنشاؤه عبر الـ CLI يحمل البيانات المعيارية التالية:
* `approval_id`: رقم معرف فريد للطلب (مثال: `APP-2026-001`).
* `action`: تفاصيل الإجراء المطلوب مراجعته (مثال: `إرسال حملة تواصل لشركة ABC`).
* `requested_by`: اسم الوكيل الذي أنشأ الطلب (مثال: `sales_agent`).
* `status`: حالة الطلب الحالية (`pending`, `approved`, `rejected`).
* `approved_by`: المالك البشري الذي اعتمد أو رفض الإجراء.
* `timestamp`: وقت وتاريخ إنشاء الطلب بالصيغة الدولية ISO 8601.

---

## 4. سياسة التعامل مع حالات التجاوز (Emergency Rollback)
عند حدوث إجراء مستقل من وكيل دون المرور بمصفوفة الموافقات المحددة، يُفعّل بروتوكول طوارئ تلقائي:
1. **التجميد التلقائي (Immediate Isolation)**: يتم سحب صلاحيات الوكيل المسبب وتعليق نشاطه فوراً في سجل الوكلاء.
2. **سجل التدقيق الحي (Audit Log Extraction)**: يستخرج محرك الحوكمة سجل الحركات الفعلي للأداة والمدخلات الفنية المعنية.
3. **الإشعار والتصحيح**: إرسال إشعار فوري لـ سامي يوضح تفاصيل الإجراء وجدول معالجة الأثر المالي في غضون 30 دقيقة.
4. **تحديث الحظر**: تحديث خوارزميات أمان MCP وتمرير الإجراء لمسار الفحص الأمني الإجباري.
