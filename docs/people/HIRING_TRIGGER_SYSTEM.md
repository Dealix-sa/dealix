# Hiring Trigger System

## Purpose
Hire only when a specific trigger is true and the operating system has the documentation to onboard quickly.

## Triggers
| Role | Trigger |
|---|---|
| Lead generation contractor | 25 leads/week become a sustained bottleneck and pipeline is the binding constraint on revenue |
| Bilingual content contractor | 2 weeks where Arabic versions of English content are blocking publication |
| Delivery operator | First retainer is signed AND founder is double-booked at least 2 weeks running |
| Finance contractor | First quarter close takes the founder > 2 days OR external accountant requires reformatted books |
| Engineering hire | After SaaS gate AND the engineer sub-agent cannot keep up with PR throughput |

## Hire-or-not algorithm
1. Is there documented evidence the trigger fired?
2. Is the role doc written and onboarding checklist ready?
3. Is the access scope defined?
4. Will the new hire's first month be measurable (4 specific outcomes)?
5. Can we afford 6 months of the role from current cash?

If any answer is No: do not hire. Document the gap and revisit next month.

## Anti-patterns
- Hiring against a feeling ("we should grow").
- Hiring before the role doc exists.
- Hiring before access can be scoped.
- Hiring before we can articulate the first 30/60/90.

## Termination
- A trial period of 90 days for every contractor.
- Performance review at days 30, 60, 90; written.
- A clean exit checklist (revoke access, archive deliverables, log in `people/access_log.csv`).
