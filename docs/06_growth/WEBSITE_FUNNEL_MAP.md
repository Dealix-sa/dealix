# Website Funnel Map — خريطة قمع الموقع

كيف يتحرّك الزائر من أي مدخل (أداة مجانية، صفحة قطاع، منشور) إلى **Free Diagnostic** أو **Command Sprint**، ثم إلى دفع ذاتي وتسليم منضبط. صفحة واحدة، CTA واحد، مسار واحد لا يقفز.

> القاعدة: لا نطلب من الزائر قرارًا أكبر مما رأى من دليل. كل صفحة تقود إلى واحد فقط من: Business OS Score · Free Diagnostic · Command Sprint.

## 2) من كل مدخل إلى Diagnostic / Sprint

```
Free Tool ──► Score ──► Top 3 Weaknesses ──► Recommended OS ──► Free Diagnostic
Sector Page ──► Sector Diagnostic ──► Command Sprint
Sample Pack ──► "أريد مثله لشركتي" ──► Command Sprint
Proof Library ──► ثقة ──► Free Diagnostic
Academy ──► Operating Rhythm ──► Command Sprint
Template ──► استخدام ──► "أحتاج تشغيله" ──► Free Tool ──► Score
```

كل مسار ينتهي عند نقطة قرار صغيرة (Diagnostic مجاني) أو قرار محدود المخاطر (Sprint بسعر مؤسس)، لا عند عقد سنوي.

---

## 3) تدفق الدفع الذاتي (Self-serve Checkout)

```
Choose Sprint ──► Pay (Moyasar) ──► Intake ──► Start Date ──► Customer Folder ──► Founder Review ──► Delivery
```

- **Choose Sprint:** 499 أو 1,500 SAR — السعر معروض بوضوح، لا مفاجآت.
- **Pay:** بوابة موثّقة فقط (Moyasar live). لا تنفيذ تلقائي خارجي.
- **Intake:** نموذج قصير (الأهداف، القطاع، مصادر البيانات، نقاط الألم).
- **Start Date:** يختار العميل تاريخ بدء ضمن الطاقة المتاحة (scarcity صادقة — التسليم يدوي).
- **Founder Review:** كل تسليم يمرّ بمراجعة بشرية قبل أن يصل العميل (Approval-first).

---

## 4) ما يتولّد تلقائيًا عند الدفع

| المُولَّد | المحتوى |
| --- | --- |
| Customer Folder | مجلد عميل منظّم (intake، مخرجات، proof) |
| Payment Record | سجل دفع موثّق (المبلغ، التاريخ، العرض) |
| Intake Checklist | قائمة ما نحتاجه من العميل قبل البدء |
| Delivery Timeline | جدول 7 أيام بمعالم واضحة |
| Proof Pack Skeleton | هيكل Proof Pack جاهز للتعبئة أثناء التسليم |
| Upsell Task | مهمّة مؤجّلة: عرض Managed **بعد** Proof فقط |

> ملاحظة امتثال: لا بيانات عميل تُستخدم للتدريب، ولا اسم يُنشر بدون موافقة. الـ Upsell Task لا يُفعّل قبل Proof Pack.
