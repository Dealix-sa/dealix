# Dealix Design OS Full Operating Model

## Purpose

This model explains how Dealix should use the Design Command Room OS across every major company function.

## The universal loop

```text
Signal -> Brief -> Artifact -> Validation -> Review -> Handoff -> Proof
```

## Company-wide application

| Function | Input signal | Artifact | Review gate | Handoff |
|---|---|---|---|---|
| Founder/CEO | daily risks, revenue blockers, delivery blockers | Founder War Room | decision review | founder next action |
| Revenue | accounts, follow-ups, proposals, approvals | Revenue Command Room | outbound safety | sales action queue |
| Marketing | offer, ICP, pain, proof | Landing Page | claims review | web preview / production brief |
| Sales | prospect pain, offer, pilot scope | Sales Deck | claims and pricing review | PDF/PPTX workflow |
| Delivery | intake, pain, workflow, risks | Delivery OS | scope review | delivery sprint |
| Client proof | diagnosis, actions, validation | Proof Pack | evidence review | client review |
| Growth operations | channel tasks, replies, segments | Client Growth | channel safety | approval cards |
| Company Brain | signals, metrics, assumptions | Company Brain artifact | executive review | decision backlog |
| Product/frontend | command room needs, UX gaps | HTML preview / UI brief | production handoff | app implementation |

## Review layers

1. **Structural validation** — generated by `make design-os-validate`.
2. **Claims review** — check fake/unsupported claims.
3. **Safety review** — check external-send implication and sensitive data.
4. **Business review** — check if the artifact drives a real next action.
5. **Engineering review** — only needed before production UI.

## Daily operating command

```bash
make design-os-daily CONTEXT="Dealix founder day: revenue, delivery, product, trust, and next actions"
```

## Weekly operating command

```bash
make design-os-all CONTEXT="Dealix weekly commercial and delivery review"
make design-os-index
make design-os-validate
make design-os-html
```

## Promotion policy

Validation does not mean approval.

A generated artifact must remain internal until:

```text
Approval state:
Reviewer:
Claims status:
Safety status:
Handoff target:
Production/client decision:
```

are explicitly filled and accepted.
