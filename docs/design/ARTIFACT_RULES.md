# Dealix Design Artifact Rules

## Purpose

These rules govern agent-generated Dealix pages, decks, dashboards, proof packs, prototypes, and visual reports.

The goal is to gain the useful Open Design workflow — design system + skill + artifact + critique + handoff — without importing a separate desktop product or breaking Dealix's existing app architecture.

## Artifact classes

| Class | Examples | Default status |
|---|---|---|
| Prototype | dashboard mock, landing page mock, mobile flow | `draft` |
| Deck | sales deck, investor deck, proposal deck | `needs_review` |
| Client artifact | proof pack, diagnosis page, workflow map | `needs_review` |
| Internal report | founder war room, revenue report, delivery report | `draft` |
| Production UI | app route, component, dashboard screen | `requires_approval` |

## Required metadata block

Every artifact must begin with or include:

```text
Artifact:
Owner:
Business goal:
Primary user:
Source context:
Generated by:
Approval state:
Safety status:
Handoff target:
```

## Approval states

```text
draft
needs_review
approved_for_demo
approved_for_client
approved_for_production
rejected
```

## Handoff targets

```text
internal_review
client_review
sales_use
web_production
app_production
pdf_export
pptx_export
html_preview
```

## Safety gates

Before any artifact is approved for client or production use, check:

- no fake customer names
- no fake testimonials
- no fake logos
- no guaranteed revenue or guaranteed ROI
- no claims of official partnership without proof
- no implied live outbound unless live gates exist
- no sensitive data exposure
- no credentials or tokens
- no unlicensed third-party asset copy

## Accessibility baseline

Generated UI or HTML prototypes should:

- use semantic headings
- preserve text contrast
- avoid text inside images when possible
- include descriptive labels for key controls
- be keyboard-readable where practical
- avoid tiny dense text for executive screens

## Brand baseline

Artifacts should follow `docs/design/DEALIX_DESIGN.md` and use Dealix's operating-system narrative:

```text
Signal -> Decision -> Action -> Proof
```

## Output placement

Use these paths unless the repo already has a more specific canonical location:

```text
reports/design/
reports/proof/
reports/command_room/
sales/
docs/design/
apps/web/app/
```

Generated temporary assets should not be committed unless intentionally part of a proof pack or release artifact.

## Review checklist

Before merging design output:

1. Does it serve a clear business goal?
2. Does it use the Dealix design contract?
3. Is the approval state visible?
4. Are claims safe and reviewable?
5. Is the output placed in the right path?
6. Is there a handoff note?
7. Did we avoid importing unnecessary runtime dependencies?
8. Did we keep outbound disabled by default?
