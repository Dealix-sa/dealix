# Dealix Design System Contract

## Purpose

This file is the canonical Dealix brand and design contract for agents, designers, frontend work, decks, proof packs, and client-facing artifacts.

It adapts the useful Open Design pattern of treating `DESIGN.md` as a brand contract, but keeps Dealix's own product identity, Saudi B2B positioning, and operating-system narrative.

## Brand positioning

Dealix is an AI-native operating system for Saudi B2B companies. It helps companies convert scattered customer, sales, delivery, and operational signals into clear daily actions, reviewed communications, proof packs, and executive decisions.

Dealix should feel:

- serious
- trusted
- sharp
- operational
- premium but practical
- Saudi/GCC-aware
- executive-grade
- not gimmicky

## Visual principles

1. **Command room clarity** — dashboards must show what matters today, not every possible metric.
2. **Proof over hype** — visual language should support evidence, logs, reports, and decisions.
3. **Human approval visible** — outbound and client-facing actions should show approval states clearly.
4. **Commercial momentum** — every product surface should push toward next action, proposal, meeting, or delivery proof.
5. **Calm authority** — avoid loud SaaS clutter, fake urgency, and over-designed decorations.

## Core product surfaces

Design work should prioritize these surfaces:

| Surface | Goal |
|---|---|
| Revenue Command Room | Daily pipeline, targets, follow-ups, proposal queue, approvals |
| Founder War Room | CEO decision desk, risks, opportunities, next actions |
| Company Brain | Signals, summaries, recommendations, executive memos |
| Client Growth Operator | Channel action cards, reply classifications, approvals |
| Client Delivery OS | Intake, diagnosis, blueprint, sprint status, proof pack |
| Trust and Safety | Outbound status, claims review, consent and risk gates |
| Website / Sales Page | Explain value clearly and convert to discovery/pilot |
| Deck / Proposal | Sell the outcome, method, pilot, and proof plan |

## Layout rules

- Use clear hierarchy: title, context, decision, evidence, action.
- Prefer one main action per screen.
- Use cards for decisions, queues, approvals, and risks.
- Avoid crowded dense dashboards unless it is a true operator view.
- Keep Arabic and English layouts symmetrical where possible.
- Use enough whitespace for executive readability.
- Every metric needs a label, source, and status.

## Copy rules

Use direct business language:

- "What changed today"
- "Next best action"
- "Needs approval"
- "Prepared, not sent"
- "Verified source"
- "Assumption"
- "Risk if delayed"
- "Proof generated"
- "Pilot scope"

Avoid:

- guaranteed ROI
- fake numbers
- unverifiable dominance claims
- buzzword-only AI language
- hidden automation claims
- aggressive outbound language

## Artifact requirements

Every generated design artifact must declare:

```text
Business goal:
Primary user:
Source data:
Approval state:
Safety status:
Handoff target:
```

## Recommended component types

- KPI card with source/status
- decision card
- approval card
- follow-up queue
- proposal queue
- risk register card
- timeline card
- proof pack section
- before/after workflow map
- channel status badge
- trust gate checklist

## Design review checklist

Before approving a Dealix artifact, check:

- Does it explain the business value in under 10 seconds?
- Does it show the next action clearly?
- Are assumptions labeled?
- Are external-send states clear?
- Are fake claims avoided?
- Is the design consistent with Dealix's executive OS positioning?
- Can this be shown to a Saudi B2B client without sounding immature?

## Production rule

Design artifacts are draft-first. Nothing generated by an agent should become production UI, a client deck, or a public claim until reviewed and approved.
