# Full-Ops — نظام التشغيل الذاتي الكامل لـ Dealix
# Dealix — The Full-Ops Autonomous Operating System

> **المبدأ الواحد:** كل شيء يجري ذاتياً — السوقنة، التصنيف، الصياغة، العروض، Proof
> Packs، التقارير، المراقبة — **حتى بوابة موافقة واحدة من المؤسس** قبل أي إرسال
> خارجي أو خصم مالي. هذه ليست شركة «نصف آلية»؛ هي شركة آلية بالكامل بحارس بشري واحد.
>
> **The one principle:** everything runs autonomously — sourcing, scoring,
> drafting, proposals, Proof Packs, reports, monitoring — **up to a single founder
> approval gate** before any external send or charge. This is not a "semi-automated"
> company; it is a fully automated company with one human guardrail.

**المرجع:** `docs/COMMERCIAL_LAUNCH_MASTER_PLAN.md` · `docs/company/AI_AGENT_ORG.md`

---

## 0. لماذا بوابة الموافقة هي المنتج — Why the Approval Gate IS the Product

يمكن تقنياً جعل الإرسال تلقائياً بالكامل. لكنّ ذلك **يكسر الشركة**:

- يُسقط اختبارات الدستور (`tests/test_doctrine_guardrails.py`) — البناء يفشل.
- يُلغي ميزة Dealix التنافسية الوحيدة: «الموافقة أولاً» + توافق PDPL. مشترٍ B2B
  سعودي يثق بـ Dealix *لأنه* لا يرسل بالنيابة عنه بلا إذن.
- يحوّل Dealix إلى أداة تواصل بارد — وهي سوق مزدحمة، رخيصة، وغير موثوقة.

لذلك: **التشغيل الذاتي الكامل = أتمتة 100% من العمل، + نقرة بشرية واحدة على
الإرسال.** الماكينة تجهّز كل شيء؛ المؤسس يضغط «اعتماد». هذا أسرع نظام مبيعات ممكن
*دون* خسارة الخندق. — *Full autonomy = 100% of the work automated + one human click
to send. This is the fastest possible sales system that still keeps the moat.*

## 1. خريطة الأتمتة — The Automation Map

لكل ماكينة: ما يجري ذاتياً، وأين تقف بوابة الإنسان.

| الماكينة | يجري ذاتياً (Autonomous) | بوابة الإنسان (Human gate) |
|---|---|---|
| **البيانات** | سوقنة من مصادر مسموحة، Source Passport، تصنيف ICP، DQ، dedup، كشف PII | — (داخلي بالكامل) |
| **المبيعات** | تأهيل 100 نقطة، توصية الدرجة، صياغة تواصل دافئ، تصيير العروض | ✋ المؤسس يعتمد كل رسالة قبل الإرسال |
| **الترويج** | تقويم محتوى، صياغة منشورات/إيميلات/GEO، جدولة قائمة النشر | ✋ المؤسس يعتمد كل أصل قبل النشر |
| **التسليم** | تشغيل Sprint الـ7 أيام، تجميع Proof Pack، تسجيل أحداث الإثبات والأصول | ✋ العميل يعتمد كل مخرج؛ المؤسس يعتمد التسليم |
| **الإغلاق** | حساب ROI، صياغة سكربتات الترقية، توليد شهادة الإتمام | ✋ ترقية فقط بعد إثبات موثّق |
| **الشراكات** | تأهيل الشركاء، بطاقة الأداء، صياغة تواصل الشركاء | ✋ المؤسس يعتمد كل عقد/تواصل |
| **المال** | تسوية Moyasar، حساب الهوامش، لوحة الإيراد | ✋ تفعيل Moyasar وأي خصم = المؤسس فقط |
| **الحوكمة** | تشغيل اختبارات الدستور، تدقيق الادعاءات، بوابة الإصدار | — (سلطة إيقاف ذاتية) |
| **المراقبة** | smoke checks، تنبيهات الصحة، تتبّع الأخطاء | — (داخلي) |

**القاعدة:** كل ما هو *داخلي* مؤتمت بالكامل. كل ما *يلمس طرفاً خارجياً أو المال*
يقف عند بوابة موافقة واحدة.

## 2. الواجهة الخلفية الذاتية — Backend Autonomy

النواة موجودة في الكود؛ التنسيق الذاتي يربطها:

- **التنسيق:** `autonomous_growth/orchestrator.py`، `auto_client_acquisition/orchestrator/`.
- **الموجز اليومي:** `scripts/dealix_pm_daily.py` → `data/daily_brief/YYYY-MM-DD.md`
  (قائمة أولويات اليوم).
- **الوحدات التشغيلية:** `data_os` · `sales_os` · `governance_os` · `proof_os` ·
  `value_os` · `capital_os` · `adoption_os` · `friction_log` · `client_os`.
- **مسار الإيراد:** `POST /api/v1/checkout` → فاتورة Moyasar → webhook موقَّع →
  سجل `payments` → حدث إثبات L5 (جاهز؛ ينتظر تفعيل الحساب فقط).
- **سجلّات لا تُمحى:** Proof Ledger · Value Ledger · Capital Ledger · Consent Ledger.

**الجدولة (Scheduler):** الحلقة اليومية/الأسبوعية تُشغَّل عبر cron / ARQ. كل دورة:
data → sales drafts → delivery follow-up → growth queue → governance audit → موجز
المؤسس. **حالة اليوم:** السكربتات موجودة وتُشغَّل يدوياً/شبه-مجدولة؛ الجدولة الآلية
الكاملة بند على خارطة الطريق (القسم 6).

## 3. الواجهة الأمامية الذاتية — Frontend Autonomy

الواجهة العامة تخدم العميل ذاتياً بلا تدخل:

- **حيّ الآن:** التشخيص المجاني الذاتي (`/diagnostic.html`)، طلب العرض → Calendly،
  طلب الشراكة، صفحات الأسعار — كلها تعمل وتعيد ردوداً ثنائية اللغة.
- **مسار الدفع:** صفحة checkout → فاتورة Moyasar مستضافة → صفحة نجاح. جاهز في الكود؛
  يُفتح لحظة تفعيل الحساب.
- **بوابة الموافقة (Approval Center):** الواجهة التي يرى فيها المؤسس كل مسودة جاهزة
  (تواصل، محتوى، عرض) ويضغط «اعتماد» أو «تعديل». هذه هي نقطة التحكم البشرية الوحيدة.

**حالة اليوم:** الصفحات العامة حيّة ومؤتمتة. لوحة الموافقة موجودة منطقياً
(`approval_center`)؛ صقل واجهتها كبند خارطة طريق.

## 4. أقوى نظام مبيعات — The Strongest Possible Sales System

المبيعات تجري كخط إنتاج آلي، خطوة واحدة فقط بشرية:

```
lead (مصدر مسموح)
  → [data] passport + ICP score          ← آلي
  → [sales] تأهيل 100 نقطة + توصية درجة    ← آلي
  → [sales] صياغة تواصل دافئ ثنائي اللغة    ← آلي
  → ✋ المؤسس: اعتماد / تعديل / رفض         ← بوابة بشرية (نقرة واحدة)
  → إرسال (بعد الاعتماد فقط)                ← آلي
  → [sales] تصيير العرض عند الاهتمام        ← آلي
  → [finance] فاتورة + checkout            ← آلي (الخصم بعد موافقة العميل)
  → [customer-success] onboarding + تسليم  ← آلي
  → [delivery] Proof Pack يوم 7            ← آلي
  → [customer-success] ترقية مشروطة بإثبات  ← آلي (العرض)، ✋ بشري (الإرسال)
```

كل صندوق «آلي» لا يحتاج بشراً. البوابة البشرية الوحيدة هي «اعتماد الإرسال» — وهي
نقرة، ليست عملاً. **هذا أسرع من أي CRM لأن العمل كله مُنجَز سلفاً؛ يبقى القرار فقط.**

## 5. أقوى نظام داخلي — The Strongest Internal System

النظام الداخلي = الوكلاء الـ12 (`docs/company/AI_AGENT_ORG.md`) + الإيقاع الأسبوعي
(`docs/operating_rhythm/`) + السجلّات التي لا تُمحى + اختبارات الدستور.

- **يدير نفسه:** الحلقة اليومية تنتج موجزاً؛ الحلقة الأسبوعية تنتج 3 قرارات.
- **يحاسب نفسه:** `dealix-governance` يوقف أي مخالفة؛ `dealix-qa` يمنع شحن الأحمر.
- **يتعلّم:** `friction_log` يغذّي المراجعة الأسبوعية لتقليل احتكاك التسليم.
- **يبرهن:** كل مشروع مدفوع يُنتج Proof Pack وأصلاً رأسمالياً — لا استثناء.

## 6. خارطة الطريق للأتمتة الكاملة — Roadmap to Full Autonomy

ما هو ذاتي اليوم، وما يحتاج عملاً (بصراحة، بلا مبالغة):

| البند | الحالة اليوم | الخطوة التالية |
|---|---|---|
| الصفحات العامة + التشخيص الذاتي | ✅ حيّ | — |
| مسار الدفع (كود) | ✅ جاهز | تفعيل Moyasar (KYC المؤسس) |
| تأهيل/تصنيف/صياغة المبيعات | ✅ يعمل | ربطه في حلقة مجدولة |
| سلسلة سجلّات الإثبات | ✅ مُصلَحة ومُدقَّقة | `alembic upgrade head` عند النشر |
| الجدولة الآلية (cron/ARQ) للحلقة اليومية | 🟡 سكربتات تُشغَّل يدوياً | جدولة cron كاملة |
| لوحة الموافقة (واجهة) | 🟡 المنطق موجود | صقل واجهة المستخدم |
| الراوترات الجذعية (10 أنظمة) | ⛔ Planned (مُعلَنة بصدق) | تُبنى عند ظهور طلب حقيقي |

**القاعدة الذهبية:** ما لم يُبنَ يُوسَم `Planned` في `dealix/registers/no_overclaim.yaml`.
لا نقول «ذاتي بالكامل» عن شيء نصف مبني. التشغيل الذاتي حقيقي بقدر ما هو مُختبَر.

## 7. الخلاصة — Bottom Line

Dealix يعمل كشركة آلية بالكامل: 12 وكيلاً متخصصاً، حلقة يومية وأسبوعية، وكل ماكينة
تجارية تجري ذاتياً. الإنسان الوحيد في الحلقة هو المؤسس عند نقرة «اعتماد» — وهذه
النقرة هي ما يجعل Dealix جديراً بالثقة في السوق السعودي. نُؤتمت العمل كله؛ نُبقي
القرار الأخير بشرياً؛ ولا نبالغ أبداً فيما بنينا.
