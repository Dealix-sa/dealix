# 12 — Agent OS — نظام الوكلاء

**الحالة — Status: `DOCS_ONLY` (العقود معرّفة كمواصفة؛ التشغيل الآلي للوكلاء `FUTURE`؛ بوّابة LLM موجودة في الريبو `BETA`)**

> لا وكيل بلا عقد. الوكيل الذي لا نعرف مدخلاته ومخرجاته وأدواته المسموحة وفئة موافقته — لا يعمل.

## الغرض — Purpose

Agent OS هو طبقة الوكلاء: الطريقة التي تتحوّل بها القدرات إلى تنفيذ مُنضبط. كل وكيل (agent) ليس صندوقًا أسود، بل وحدة محكومة بـ **عقد (CONTRACT)** صريح. الغرض أن يكون كل إجراء آلي **مرئيًا، قابلًا للإيقاف، ومربوطًا ببوّابة موافقة** قبل أي أثر خارجي.

## القيمة — Value

- يمنع الإجراء الآلي غير المحكوم — لا وكيل يتجاوز فئة موافقته.
- يجعل التكلفة والأثر مرئيين (cost tracking) لكل استدعاء.
- يضمن استمرارية الخدمة عبر **توجيه احتياطي (fallback routing)** لا يعتمد على مزوّد واحد.

## العقد المطلوب لكل وكيل — Required Agent CONTRACT

كل وكيل يجب أن يُعرّف الحقول التالية قبل التشغيل:

| الحقل | الوصف |
|---|---|
| `name` | اسم الوكيل |
| `mission` | المهمّة بسطر واحد |
| `inputs` | المدخلات المسموحة |
| `outputs` | المخرجات المتوقّعة |
| `allowed_tools` | الأدوات المسموحة (whitelist) |
| `forbidden_actions` | الإجراءات الممنوعة صراحةً |
| `approval_class` | فئة الموافقة A0–A5 |
| `logs` | سجل كل استدعاء (قبل/بعد) |
| `cost_tracking` | تتبّع التكلفة لكل نداء |
| `fallback_routing` | مسار احتياطي عند فشل المزوّد |
| `tests` | اختبارات سلوك قبل الإطلاق |
| `owner` | المسؤول البشري |
| `rollback` | كيفية التراجع |

## الوكلاء المطلوبون — Required Agents

| الوكيل | المهمّة (سطر واحد) | فئة الموافقة |
|---|---|---|
| **Market Intelligence Agent** | جمع أهداف مع `evidence_source` لكل هدف | A1 |
| **Sales Draft Agent** | صياغة مسودّات موجّهة للعميل (لا إرسال) | A2 |
| **Governance Agent** | رفض المخالفات قبل التنفيذ (الحارس) | A1 |
| **Delivery Agent** | تجهيز مجلد التسليم والمخرجات | A2 |
| **Proof Agent** | تجميع Proof Pack من تسليم حقيقي | A1 |
| **Finance Agent** | حساب الهامش وCAC/LTV ووعي الفاتورة | A4 |
| **Founder Command Agent** | عرض القرارات على المؤسس والموافقة | A3 |
| **Partner Agent** | إدارة الإحالات وحصة الإيراد بعد 3 Proof Packs | A2 |

**مبدأ بوّابة الـ LLM (LLM Gateway):** لا اعتماد على مزوّد واحد. كل نداء يمرّ عبر بوّابة توجيه تختار المزوّد، تتتبّع التكلفة، وتُفعّل المسار الاحتياطي عند الفشل.

## المُدخلات والمُخرجات — Inputs / Outputs

| المُدخل | المُخرج |
|---|---|
| عقد وكيل مُعرّف | وكيل قابل للتشغيل والمراجعة |
| نداء + مدخلات | مخرج + سجل + تكلفة |
| مخالفة سياسة | رفض من Governance Agent |

## البوابات والقواعد — Gates / Rules

- **لا وكيل بلا عقد كامل** — أي حقل ناقص = الوكيل لا يُطلَق.
- **لا إرسال تلقائي · لا واتساب جماعي · لا scraping خلف تسجيل دخول.**
- **كل وكيل بفئة A2+ يحتاج موافقة المؤسس** قبل أي أثر خارجي.
- **Governance Agent يَسبق كل إجراء** — الحارس قبل التنفيذ لا بعده.
- **لا بيانات عميل في تدريب النماذج.**
- **توجيه احتياطي إلزامي** — لا نقطة فشل واحدة في المزوّد.
