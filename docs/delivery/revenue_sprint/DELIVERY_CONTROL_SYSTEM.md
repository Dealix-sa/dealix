# Delivery Control System — Revenue Sprint

## Purpose
Guarantee that every Revenue Sprint we deliver is consistent, defensible, and reviewable.

## Principles
- One canonical deliverable shape per Sprint.
- Every Sprint passes a QA checklist before handoff.
- Delivery reports cite evidence, not opinions.
- Scope is enforced; change requests are written.

## Revenue Sprint Deliverable

### Required Artifacts
1. Qualified lead list (30–75 records).
2. Lead scoring (A/B/C priority).
3. Pain hypothesis per priority segment.
4. Arabic outreach pack (DM, follow-up 1, follow-up 2).
5. Follow-up cadence.
6. Executive memo.
7. 14-day action plan.
8. Delivery report.

### Required Metadata
- Client name and sector.
- Sprint start and delivery dates.
- Founder reviewer.
- QA reviewer (may be the founder for solo operation).
- Source list per lead.

## QA Checklist
Before handoff, confirm:
- All required artifacts present.
- All leads have source, ICP fit reason, and priority.
- Outreach pack is in Arabic + English where committed.
- Trust note appended.
- No guarantee language anywhere.
- No client data in public surfaces.

## Scope Control
- Scope is set in the proposal.
- Out-of-scope requests become change requests via `docs/delivery/CHANGE_REQUEST_PROCESS.md`.
- Change requests get a new price and a new start condition.

## Handoff
- Delivery report sent to client.
- Acceptance recorded in `dealix-ops-private/delivery/reports/`.
- Lessons learned added to the weekly review queue.

## Failure Modes (and Mitigations)
- Lead list too small → expand ICP or extend timeline; do not pad with weak leads.
- No replies → diagnose ICP and message before scaling.
- Customer asks for guarantees → reaffirm trust note; escalate to founder.
- Customer requests bespoke work mid-Sprint → change request, not silent expansion.

## Daily Delivery Gate
For each active Sprint, confirm:
- Today's deliverable item produced or blocked with a reason.
- Risk surfaced if any.
- Handoff date still realistic.
