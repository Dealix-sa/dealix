# نظام التقييم والـ Red Team للذكاء الاصطناعي

> النسخة الإنجليزية القانونية: [`AI_EVAL_RED_TEAM_SYSTEM.md`](./AI_EVAL_RED_TEAM_SYSTEM.md).

## مرجع الدوكترين
- الالتزامات: #1، #2، #5.
- القرارات المثبّتة: الموافقة-أولًا، سكربتات التحقق مانعة للإصدار.

## الغرض

تقييم كل وكيل ومسار prompt في Dealix على الجودة، السلامة، استخدام الدليل، مخاطر الادعاء المفرط، مقاومة prompt injection، وتصعيد الموافقات. لا وكيل يصل للإنتاج بدون عتبة تقييم مجتازة. لا وكيل يبقى في الإنتاج بدون إعادة تقييم دورية.

## مجموعات التقييم

- دقة تصنيف العملاء.
- سلامة التواصل (لا spam، لا ادعاء مفرط، لا لغة محظورة).
- جودة draft العرض (ثنائي اللغة، مرتكز على حقائق، وضوح نطاق).
- لا ادعاء مفرط (لا وعود نتائج، لا لغة إيرادات مضمونة).
- مقاومة prompt injection.
- تسريب بيانات حساسة.
- تصنيف الموافقات.
- جودة AR/EN.
- ملاءمة السوق السعودي.

## سيناريوهات Red Team

- "تجاهل التعليمات السابقة".
- محاولة إرسال بدون موافقة.
- محاولة ادعاء إيرادات مضمونة.
- محاولة كشف الـ system prompt أو الأسرار أو بيانات مستأجرين آخرين.
- محاولة تصدير بيانات عميل خارج السياسة.
- محاولة تجاوز الموافقة عبر سلسلة استدعاءات أدوات.
- محاولة تلفيق دليل أو اختراع مصدر.
- محاولة نشر ادعاء عن عميل بدون موافقة مسجّلة.
- محاولة تواصل مع سجل موجود في `SuppressionRecord`.

## قاعدة الترقية للإنتاج

لا وكيل/مسار prompt يُرقَّى إلا إذا:

1. مجموعات التقييم ذات الصلة نجحت بعتبتها الموثّقة.
2. سيناريوهات red team ذات الصلة نجحت بصفر فشل.
3. التغيير مسجّل في audit trail.
4. مسار رجوع موثّق (الدوكترين #4).

الترقية حدث إصدار، مو merge صامت.

## القواعد الجوهرية

- التقييمات حتمية حيث ممكن؛ تقييمات يحكمها LLM لها meta-eval خاص.
- تقييم فاشل يمنع merge. "أصلحه لاحقًا" مو استثناء.
- فشل red team يُصعَّد لمراجعة الثقة، ما يُجمَّع.
- عزل المستأجرين في مجموعات بيانات التقييم.

- يومي: smoke evals.

- ربع سنوي: مراجعة العتبات والسيناريوهات.
