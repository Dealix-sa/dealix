# Before / After Template

> **Status:** Used in proposals and content. Real numbers, real clients (with permission).
> **Companion:** `PROOF_PACK_TEMPLATE_AR.md` + `CASE_STUDY_TEMPLATE_AR.md`.

## The structure (3 columns: Before, After, Why)

| Dimension | Before | After | Why it changed |
| --- | --- | --- | --- |
| Time to first reply | 18 hours | 4 hours | Routing + tags |
| Missed follow-ups per week | ~10 | ~1 | Daily digest |
| Founder time on pipeline review | 2 hours/day | 15 min/day | Command Center |
| Proposal prep time | 4 hours | 30 min | Proposal OS |
| Lost leads per month | ~20 | ~3 | Audit + WhatsApp OS |

## The hard rules

1. **Numbers must be real.** "10x improvement" without a baseline is a banned claim.
2. **Baseline must be measurable.** "We used to lose leads" is not a baseline; "we lost 20 leads/month before the audit" is.
3. **Time period must be stated.** "After 4 weeks" not "After we did the work".
4. **Client must have signed off** on the numbers.
5. **Anonymize by default.** Use the structure; redact the numbers if the client didn't agree.

## The 3 "before" patterns to avoid

1. **"We had no system"** — too vague. What specifically didn't work?
2. **"It was chaos"** — what does chaos look like in numbers?
3. **"Sales was slow"** — slow compared to what?

## The 3 "after" patterns to avoid

1. **"We 10x'd revenue"** — that's a guarantee, not a measurement.
2. **"AI solved everything"** — overclaim. AI-with-humans.
3. **"It was easy"** — never use "easy" in B2B.

## The "why it changed" column

This is the most important column. It explains:

- Which intervention caused the change.
- The mechanism (not just the outcome).
- The trade-off (what we did NOT do).

Example:

> **Why it changed:** The WhatsApp routing rules + the daily digest + the founder's daily review of stuck deals. We did NOT auto-reply. We did NOT replace the sales team. We added a layer.

## The proposal version

In a proposal, the before/after table is the centerpiece of the "current leakage" + "proposed system" sections. The table is the proof that the engagement is worth the price.

## The content version

In a LinkedIn post or a 1-page case study, the table is the visual hook. Use 1 row, not 5. Make it the row that matters.

## The trust preflight connection

The before/after numbers are checked by the trust preflight:

- If the "before" number has no source, the draft is rejected.
- If the "after" number is suspiciously round (exactly 2x, exactly 10x), the draft is flagged.
- If the time period is missing, the draft is rejected.

## When to update

- Every new client engagement = new before/after table.
- Every renewal = refresh the table.
- Every content piece = pull 1 row from the latest table.
