# Delivery Operating System

How every engagement is delivered, by tier. Approval-first throughout; the
system never communicates externally on the client's behalf.

## Per-offer delivery contract

| Offer | Inputs from client | Dealix builds | Dealix does NOT build | Timeline | Acceptance criteria |
|-------|--------------------|----|----|----|----|
| Audit | One workflow walk-through | Workflow map, bottleneck + risk diagnosis, pilot rec | Software, integrations | 3–7 d | Client agrees the map reflects reality |
| Pilot | Limited dataset, 1 workflow owner | Working workflow + approval gates | Multi-dept rollout | 14–30 d | Measurable workflow output meets agreed metric |
| Department OS | Dept data + SME access | Production workflow OS | Other departments | 30–90 d | Dept runs on the OS; handover accepted |
| Retainer | Ongoing access | Monitoring, improvements, reporting | New dept builds | Monthly | Monthly report + agreed improvements shipped |
| Enterprise | Multi-workflow scope, integration access | Multi-workflow OS + governance | Anything outside SOW | Custom | SOW milestones accepted |

## Cross-cutting boundaries
- **Security boundary:** least-privilege access; data minimization; NDA/DPA first.
- **Human approval boundary:** every external action is founder/client-approved.
- **Success metrics:** defined up front as targets, never guaranteed.
