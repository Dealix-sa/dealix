# SOP تسليم التشخيص عبر مهارة improve — ديليكس

**الغرض:** إجراء تشغيلي متكرّر يحوّل مهارة `improve` (`.claude/skills/improve/`)
إلى **محرّك تسليم** لعروض التشخيص الموجودة أصلاً (التشخيص المجاني →
Transformation Diagnostic Sprint → Managed Ops). الهدف: نفس الجودة، تكلفة أقل،
تسليم قابل للتكرار.

**الخطوط الحمراء (غير قابلة للتفاوض):**
- كل مخرَج داخلي (A1) حتى يراجعه المؤسس (سامي عسيري) ويصير مسودّة عميل (A2).
- لا إرسال خارجي، لا تنفيذ تلقائي، لا دمج بدون موافقة العميل.
- بيانات العميل الحسّاسة/PII/الأسرار **لا** تذهب لأي مزوّد مجاني. كود/أنظمة العميل
  فقط، وبموافقته. المستشار (Opus) والمنفّذ الرخيص للأعمال غير السرّية.
- لا أرقام ملفّقة، لا ضمانات. لغة فرضية: «نتوقّع» / «سنقيس».

## مصفوفة العروض ↔ أوامر improve ↔ التسليم

| العرض (السلّم الرسمي) | السعر | أمر improve | المخرَج | يملأ الملف |
|---|---|---|---|---|
| التشخيص المجاني | مجاني | `/improve quick` | جدول نتائج مرتّب (top 3–5) | `customers/<name>/02_diagnostic_summary.md` |
| Micro Sprint | 499 ريال | `/improve <category>` | 2–3 نتائج + خطة واحدة | `02_diagnostic_summary.md` + `plans/` |
| Data Pack | 1,500 ريال | `/improve` على مسارات البيانات | تقرير أدلة DQ | قالب تقرير التشخيص |
| Diagnostic Sprint | 7,500–25,000 ريال | `/improve deep` | تدقيق كامل + مجموعة خطط + Proof Pack | `plans/INDEX.md` + Proof Pack |
| Managed Ops | 2,999–4,999 ريال/شهر | `/improve reconcile` دوري | backlog تحسين حيّ | `plans/` عبر الزمن |

> السعر النهائي والنطاق يُحدَّدان بعد جلسة التشخيص المجانية، وفق السلّم الرسمي في
> `CLAUDE.md`. لا تُستخدم أرقام أي ملف تسعير قديم.

## المسار التشغيلي (خطوة بخطوة)

### 1) التأهيل (قبل أي تشغيل)
`improve` يناسب العميل إذا عنده **أنظمة رقمية فعلية**: كود، أدوات داخلية،
تدفّقات بيانات، أو تكامل. إن كان العميل بلا أي نظام رقمي، هذا ليس العرض المناسب —
حوّله لعرض غرفة القيادة التشغيلية بدل تدقيق الأنظمة. كُن صادقًا في الملاءمة.

### 2) الاستطلاع (Recon)
- شغّل `/improve` على نطاق مُتفَّق عليه فقط (repo/نظام محدّد).
- تأكّد من نضارة رادار المزوّدين قبل أي تنفيذ رخيص: `make ai-provider-registry-check`.
- كل نتيجة يجب أن تحمل دليل `file:line` أو مصدر بيانات. لا نتائج عامة.

### 3) التدقيق + التحقّق (A1 داخلي)
- المستشار (Opus) يدقّق ويعيد قراءة كل دليل قبل الإدراج (خطوة vet).
- سجّل النتائج المرفوضة مع السبب (شفافية = ثقة).
- انسخ الجدول إلى `02_diagnostic_summary.md` للعميل.

### 4) بوابة مراجعة المؤسس (A1 → A2)
- المؤسس يراجع النتائج قبل أي عرض للعميل. هنا يتحوّل التحليل الداخلي إلى مسودّة عميل.
- لا يخرج شيء بدون هذه البوابة.

### 5) التسليم
- **مجاني:** جدول النتائج فقط = الطُّعم الذي يفتح الترقية.
- **Sprint:** مجموعة خطط قابلة للتنفيذ (`plans/NNN-*.md`) + Proof Pack.
- استخدم `sales/DIAGNOSTIC_REPORT_TEMPLATE_AR.md` كغلاف التسليم.

### 6) التنفيذ (Sprint/Enterprise فقط، بموافقة العميل)
- وكيل `improve-executor` ينفّذ خطة واحدة في worktree معزول، يشغّل بوابات الخطة.
- المؤسس/العميل يراجع الفرق (diff). **الدمج قرار العميل** — لا دمج تلقائي.

### 7) الترقية (Upsell)
- التشخيص المجاني → Sprint: «حوّلنا النتائج إلى خطط منفَّذة + إثبات».
- Sprint → Managed Ops: «`/improve reconcile` شهريًا يبقي التحسين حيًّا».

## اقتصاديات الهامش (لماذا هذا مربح)

- **المستشار (Opus):** ساعة تحليل غالية تنتج مجموعة خطط.
- **المنفّذ:** موديل من الطبقة المجانية (عبر رادار المزوّدين) ينفّذ الأعمال غير السرّية.
- الفجوة بين قيمة الحكم (مفوتر) وتكلفة التنفيذ (شبه صفر) = هامش Sprint.
- التكرار عبر `reconcile` يحوّل الصفقة لمرّة واحدة إلى إيراد متكرّر (Managed Ops).

- [ ] كل نتيجة لها دليل (`file:line`/مصدر). لا أرقام مخترعة.
- [ ] النتائج المرفوضة مُسجّلة بأسبابها.
- [ ] بوابة مراجعة المؤسس تمّت (A1 → A2).
- [ ] لغة فرضية فقط، بلا ضمانات.
- [ ] لا بيانات عميل حسّاسة ذهبت لأي مزوّد مجاني.
- [ ] الدمج بموافقة العميل الصريحة فقط.
