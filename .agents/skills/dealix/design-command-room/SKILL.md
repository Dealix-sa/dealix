# Dealix Design Command Room OS

## When to use

Use this skill when the user asks to:

- design a Dealix dashboard, command room, deck, landing page, or proposal
- improve Dealix UI/UX
- create a frontend prototype
- create a client proof pack page
- create a sales or investor presentation structure
- convert a business brief into a visual artifact
- apply Open Design-style agentic design workflow inside Dealix

## Core doctrine

Dealix design work should follow:

```text
brief -> design contract -> artifact -> critique -> approval -> handoff
```

Design is not decoration. It must help the founder, operator, or client make a decision and take the next action.

## Dealix design language

Use `Signal -> Decision -> Action -> Proof` as the default information architecture.

Core sections:

1. what changed
2. why it matters
3. what to do next
4. what needs approval
5. what proof exists
6. what risk remains

## UX rules

- Prefer executive clarity over visual complexity.
- Do not create crowded vanity dashboards.
- Put the next action above secondary analytics.
- Show approval states clearly.
- Distinguish draft, prepared, sent, verified, and blocked.
- Every metric needs a source/status.
- Every claim needs proof or an assumption label.
- Do not use fake client logos or fake testimonials.

## Frontend rules

When editing `apps/web`:

- inspect existing routing and components before adding new files
- reuse existing design tokens and components if present
- avoid adding new UI dependencies unless necessary
- keep generated prototype routes separate from production routes until approved
- add validation notes to the PR body
- do not break existing build commands

## Output placement

Use:

```text
reports/design/
reports/command_room/
reports/proof/
sales/

apps/web/app/
```

Do not commit temporary generated image/video files unless intentionally part of a reviewed proof pack.

## Safety gates

Before a design artifact becomes client-facing or production-facing, verify:

- no fake ROI
- no fake testimonials
- no fake customer logos
- no guaranteed revenue
- no misleading automation claim
- no sensitive data
- no live outbound implication unless gates exist
- approval state is visible

## Definition of done

A Dealix design artifact is complete when:

- it follows `docs/design/DEALIX_DESIGN.md`
- it has metadata
- it has approval state
- it supports a real business decision or sales/delivery action
- unsafe claims are removed
- the handoff target is clear
- production use is gated
